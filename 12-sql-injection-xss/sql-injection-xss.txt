After successfully setting up dvwa, I began testing its vulnerable modules. I first accessed the sql injection feature and entered a user id of 1. The query returned the administrator’s credentials, confirming that the application was susceptible to sql injection attacks. Next, I moved on to the xss module. In the input field, I typed the payload which triggered a javascript alert in the browser. This demonstrated that dvwa was vulnerable to cross‑site scripting. Finally, I tested session handling by logging in with valid credentials and then using the browser’s back and forward buttons. I observed that the application did not enforce proper session expiration, allowing me to remain authenticated even after navigating away. This behavior highlighted a broken authentication vulnerability within dvwa.
